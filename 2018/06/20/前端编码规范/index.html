前端编码规范

发表于 2018-06-20 | 更新于: 2018-06-20 | 分类于 杂谈 |
字数统计: 1,929 | 阅读时长 ≈ 8

一、HTML规范

1.1 用两个空格来代替TAB键–这是唯一能保证在所有环境下获得一致展现的方法。

1.2 嵌套元素应当缩进一次（即两个空格）。

1.3 为 html 根元素指定 lang 属性，从而为文档设置正确的语言。这将有助于语音合成工具确定其所应该采用的发音，有助于翻译工具确定其翻译时所应遵守的规则等等。

1.4 IE 支持通过特定的 标签来确定绘制当前页面所应该采用的 IE 版本。除非有强烈的特殊需求，否则最好是设置为 edge mode，从而通知 IE 采用其所支持的最新的模式。

1.5 通过明确声明字符编码，能够确保浏览器快速并容易的判断页面内容的渲染方式。

1.6 尽量遵循 HTML 标准和语义，但是不要以牺牲实用性为代价。任何时候都要尽量使用最少的标签并保持最小的复杂度。

二、CSS规范

CSS书写规范

2.1 用两个空格来代替制表符（tab） – 这是唯一能保证在所有环境下获得一致展现的方法。

2.2 为选择器分组时，将单独的选择器单独放在一行。

2.3 在每个声明块的左花括号前添加一个空格。

2.4 每条声明语句的 : 后应该插入一个空格。

2.5 对于属性值或颜色参数，省略小于 1 的小数前面的 0 （例如，.5 代替 0.5；-.5px 代替 -0.5px）。

2.6 尽量使用简写形式的十六进制值，例如，用 #fff 代替 #ffffff。

2.7 避免为 0 值指定单位，例如，用 margin: 0; 代替 margin: 0px;。

2.8 声明顺序（如下）

相关的属性声明应当归为一组，并按照下面的顺序排列：
1. Positioning（位置）
2. Box model（盒模型）
3. Typographic（文字）
4. Visual（外观）

由于定位（positioning）可以从正常的文档流中移除元素，并且还能覆盖盒模型（box model）相关的样式，因此排在首位。盒模型排在第二位，因为它决定了组件的尺寸和位置。其他属性只是影响组件的内部（inside）或者是不影响前两组属性，因此排在后面。

2.9 对于只包含一条声明的样式，为了易读性和便于快速编辑，建议将语句放在同一行。对于带有多条声明的样式，还是应当将声明分为多行。

2.10 简写形式的属性声明(如下)

注意以下几个属性：
1. padding
2. margin
3. font
4. background
5. border
6. border-radius

大部分情况下，我们不需要为简写形式的属性声明指定所有值。例如，HTML 的 heading 元素只需要设置上、下边距（margin）的值，因此，在必要的时候，只需覆盖这两个值就可以。**过度使用简写形式的属性声明会导致代码混乱，并且会对属性值带来不必要的覆盖从而引起意外的副作用**。

2.11 class命名只能出现小写字符和破则号-，不可使用下划线或者驼峰形式命名

CSS重置

CSS命名规范

初步采用B-E-M命名规范：

  1. Block（块）：页面的某个单元。它可以是一个基础的单元，也可以是包含其它基础单元的复合单元。
  2. Element（元素）：元素是块的一部分。元素是依赖上下文的——它们只有处于他们应该属于的块的上下文中时才是有意义的。
  3. Modifier（修饰符）：对元素的描述。比如某个tab选项卡，激活与未激活时的状态是不一致的，修饰符就是对元素激活时状态的描述。

对于块和元素的命名，一般使用名词，对于修饰符的命名，一般使用形容词。可参考以下内容：

  • 头：header
  • 内容：content
  • 尾：footer
  • 导航：nav
    • 主导航：nav-main
    • 子导航：nav-sub
  • 菜单：menu
    • 主菜单：menu-main
    • 子菜单：menu-sub
  • 侧栏：sidebar
  • 栏目：column
  • 页面外围控制整体佈局宽度：wrapper
  • 左右中：left right center
  • 页面主体：main
  • 列表：list
    • 列表项：list-item
  • 面包屑：crumb

移动端布局统一解决方案

由于目前主流浏览器均已支持VW（包括IE11），因此可以放心在项目中使用VW布局来代替REM布局，具体参考在VUE项目中使用VW布局

三、统一规范

目的：提高效率和降低成本。这里说的效率和成本，并不只是公司层面，还包括个人。良好的工程化，能降低沟通成本，实现更好的协同，节省开发和测试人员的重复劳动，降低发布的常见问题等等，经过有效实践，工程化的推广还能极大地减少加班的时间。

1. 规范

  • 文档：完整的需求文档
  • 技术选型：
    • JS框架：Vue
    • CSS模块化方案：stylus
    • 打包及自动化：webpack
    • 代码管理：Git
  • 代码规范：见上文HTML与CSS部分（暂时不对javascript编码进行规范）

2. 协作

  • 设计稿及需求文档确定后组织讨论、分工。
  • 提取页面公共模块（多个页面中相同或相似部分）。
  • 每个人都要了解页面公共模块，公共模块由某个人开发后交付他人使用。
  • 按页面类型分配，同一类型页面分配给同一个人。
  • 写静态页面的人仍然需要了解需求后才能开始写页面。否则静态页面写完后看着没问题，实则无法满足交互需求而重新修改导致浪费时间。

3. 创新

  • 发掘其他团队的优秀方法
  • 多交流，多观察业界
